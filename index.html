01 Executive Summary

DeepFlow's core value proposition is clear and differentiated: give the system a prompt, and it generates a structured task graph with dependency, parallelism, and multi-level decomposition. The graph is the product. This review assesses how well the current UI serves that mission — and where it can improve.

🏗️

Strong Foundation

Clean visual language, multi-view system (canvas, kanban, list, swimlane, dashboard), distinctive brand colour, and a well-conceived workspace for human-AI collaboration. The architecture is sound.

🔍

Graph Needs Clarity

Nodes are visually identical regardless of status. Edges don't distinguish dependency types. Text is truncated everywhere. The graph's power is hidden behind uniform blue circles and dotted lines.

🧭

Navigation is Opaque

Multi-level drill-down (L1→L2→L3) lacks breadcrumbs, clear entry/exit points, and familiar metaphors. The 3D overview is visually striking but interaction-free.

🗣️

Language Barrier

Technical jargon throughout — "Task Execution Model", "L1/L2/L3", "Agents", "Failed" — assumes knowledge the target audience doesn't have. Business users need human words.

⚡

Quick Wins Available

Many high-impact improvements are low-effort: tooltips, breadcrumbs, label replacements, status-as-node-colour. A focused sprint could dramatically improve approachability.

🏛️

Governance Ready

The ICAAP workflow demonstrates real enterprise use: named assignees, audit trails, board approval gates. The platform is governance-capable — the UI needs to surface this more clearly.

💡 The Core Insight

DeepFlow doesn't need to be simpler. It needs to explain itself — with familiar shapes, human language, and a clear on-ramp to genuine power. The graph should feel like a superpower, not a prerequisite.

02 What Works Well

Before diving into improvements, it's important to acknowledge what DeepFlow does right — there's a lot to build on.

🎯 Prompt-to-Graph Generation

No mainstream PM tool generates dependency-aware, parallelism-optimised task graphs from natural language. This is a genuine moat and the killer feature. The value proposition is exciting — if users can understand what they're looking at.

🌈 Depth-Colour Gradient System

DeepFlow uses a blue→red colour gradient based on hierarchy depth. L1 (strategic objectives) = blue/cyan, deepest levels (atomic tasks) = pink/red, with intermediate levels blending between. This applies to nodes, badges, connection lines, and the 3D navigator layers — a cohesive visual language that communicates "where you are" in the hierarchy at a glance. This is a genuinely sophisticated design decision that most PM tools lack entirely.

⬡ Shape-Based Task Encoding

Circles = human tasks, hexagons = AI agent tasks. This lets users scan the graph and instantly see the human/AI balance without reading labels. Combined with the depth-colour gradient, DeepFlow achieves two independent visual dimensions: shape encodes WHO (human vs AI), colour encodes WHERE (depth level). Both survive at any zoom level — even when nodes are tiny dots, circles and hexagons remain distinguishable. This also maps naturally to the "Users vs Agents" analytics in the Level Hub.

👁️ Multi-View System

Canvas, kanban, list, swimlane, and dashboard views give users multiple ways to interact with the same underlying graph. This is genuinely impressive view diversity that respects different working styles.

🎨 Clean Visual Foundation

The cyan/sky-blue accent is distinctive and well-chosen. Dark sidebar provides excellent anchoring. Status colour coding (green/amber/red/grey) is intuitive and consistent across all views.

🤖 Human-AI Workspace

The three-panel workspace (output + chat + details) is the right pattern for human-AI collaboration. It features expandable AI thought process steps, version navigation (V3/3), an "Execute this task for me" button, a History tab with timestamped audit trail, and a Resources tab with inputs/outputs. This is genuinely powerful task execution infrastructure.

🏠 Inviting Home Screen

"Hello Ali!" with the prompt bar, project cards with graph thumbnails, and "My Tasks" sidebar — this is a warm, approachable entry point that balances exploration and quick access.

📊 Level Hub Analytics

The donut chart, task due list, and Users vs. Agents breakdown per level means stakeholders can monitor progress through familiar dashboard widgets without touching the canvas.

03 The Graph — Making the Hero Shine

The graph is DeepFlow's entire differentiator. Every recommendation here is oriented around one question: how do we make the graph feel as natural as a Monday.com board while preserving the power that makes DeepFlow unique?

✅ What Already Works — Node Shape & Colour System

DeepFlow already has two strong visual encoding systems: circles for human tasks (with avatar/initials) and hexagons for AI agent tasks (with agent logo/letters), plus a depth-colour gradient (blue/cyan at L1 → pink/red at deepest levels) applied to node borders, connection lines, and badges. Shape encodes WHO, colour encodes WHERE — two independent dimensions that work well together. The recommendations below build on this foundation rather than replacing it.

Critical Node status is still under-communicated UXUIBusiness

Screens: Workflow Canvas, Swimlane, Multi-level Drilldown

The circle/hexagon shape distinction and depth-colour gradient are strong foundations. However, task status is still communicated only through a small badge overlay — easy to miss at working zoom levels. Users should understand workflow state in under 3 seconds without squinting at individual badges.

💡 Recommendation Add status as a secondary visual layer that works alongside the depth-colour and shape systems. Options: (1) Use the node border style — solid for complete, dashed for in-progress, dotted for pending, double for failed. (2) Add a status glow/ring outside the depth-colour border — green glow for done, amber for active, red pulse for failed. (3) Use node opacity — completed tasks at 60% opacity, creating a "fading into the background" effect that makes active work pop. The key: status encoding must not conflict with the depth-colour system — it should be a third independent dimension.

Critical Edges don't communicate dependency type UIBusiness

Screens: Workflow Canvas, Node Closeup

All edges use the same dashed cyan line regardless of whether they represent sequential dependency, parallel branching, or convergence. Small chevron arrows are nearly invisible (light cyan on white, ~8px). The graph's entire value is showing which tasks depend on others — if every edge looks the same, the graph is just a pretty picture.

💡 Recommendation Use solid lines for sequential dependencies, dashed for optional/conditional paths. Colour-code by status: green (satisfied), amber (in progress), red (broken), grey (pending). Add animated dot-flow along edges to show direction. Increase arrowheads to 12px+ at the target node. Add fork/join indicators where parallel branches diverge and converge.

Critical Universal text truncation destroys task identification UXUIBusinessFrontend

Screens: All views

Nearly every task label across the entire application is truncated. When every label is a sentence fragment, the information architecture has failed. Task names are the content of a task management tool.

💡 Recommendation Allow 2-line wrapping on canvas nodes with ellipsis + full-text tooltip on hover. Never truncate in headers or detail views. Auto-collapse empty columns in list view to give more space to titles. Consider encouraging shorter task names via UI hints, or auto-generating a short title + full description.

Critical Graph layout lacks spatial grammar UIBusiness

Screen: Workflow Canvas

The current layout appears force-directed with no clear reading direction. Business users expect a process map that reads like a timeline — not a physics simulation. The ICAAP workflow demonstrates that left-to-right flow works brilliantly when enforced.

💡 Recommendation Default to left-to-right flow with clear start and end columns. Vertically align parallel tasks within the same stage. Add start/end markers. Consider grouping tasks into visual phases with subtle background shading. Offer a "fit to readable" smart zoom that optimises for label legibility.

High Ensure shape parity between circles and hexagons UIUX

Screens: Workflow Canvas, Node Closeup

The circle/hexagon distinction is excellent. Two refinements: (1) Hexagons are visually "heavier" than circles at equal size — AI tasks may inadvertently draw more attention, subordinating human tasks visually. (2) At far zoom, both shapes should remain distinguishable — circles as round dots, hexagons retaining their angular silhouette.

💡 Recommendation Calibrate both shapes to have equal visual weight — slightly increase circle border thickness or size to match hexagonal solidity. At far zoom, ensure hexagons don't collapse into circles (maintain angular rendering down to ~8px). When no agent logo is assigned, use a subtle circuit/bot icon inside hexagons rather than generic initials. Consider adding a thin "layered squares" icon on nodes that have sub-levels, signalling drill-down availability.

High Undiscoverable node interactions UXUI

Screen: Node Closeup (radial menu)

The radial context menu appears via an unknown trigger with no visual hint. Editing the graph — adding connections, removing nodes — is fundamental. The radial menu itself uses a unique visual language that shares zero DNA with any other component.

💡 Recommendation On hover, show a subtle "…" overflow icon on both circles and hexagons. Offer a standard right-click context menu with icon + text labels as the primary path. Keep the radial menu as a power-user shortcut. Add a first-use coach mark: "Click a task to open it. Right-click for more actions."

High No way to add nodes on the canvas UX

Screen: Workflow Canvas

The kanban view has "+ Add Task" buttons in every column. The canvas — the flagship view — has no visible affordance for adding new tasks.

💡 Recommendation Add a "+ Add Task" button in the canvas toolbar. Allow double-click on empty canvas to create a node. Consider a drag-from-edge affordance: hover on a node's right side → small "+" connector handle → drag to create a connected node (like Miro/FigJam).

High No critical-path or progress overlay UIUXBusiness

Screen: Workflow Canvas

With tasks in various states, the workflow's health is invisible at a glance. Downstream impact of failures is not shown.

💡 Recommendation Add a progress summary bar at the top: "2/5 complete · 1 blocked · ETA: stalled". Offer a critical-path highlight mode that dims non-critical tasks and emphasises the longest dependency chain. When a task fails, visually grey out downstream edges to show cascade impact.

High Dependency intelligence missing from other views UXBusiness

Screens: Kanban, List View

Dependencies — the graph's unique value — are invisible in kanban and list views. A blocked task looks identical to an unblocked one. The graph's killer insight doesn't surface where business users spend most of their time.

💡 Recommendation Kanban cards: Add "Blocked by: [Task Name]" chip or 🔗 icon with count. List view: Add "Depends on" and "Blocks" columns. Both views: Flag tasks on the critical path. The graph's data should be everywhere, even when the graph's visual isn't.

04 Multi-Level Navigation

The L1 → L2 → L3 decomposition is DeepFlow's most distinctive feature. The challenge is making it intuitive for users who think in folders and tabs, not graph theory.

The "Semantic Zoom" Model

Users already understand zooming: pinch to zoom out (satellite view), zoom in (street view). Apply this to levels:

  • Overview state: L1 nodes shown as large group containers with summary info (task count, completion %, assignees)
  • Double-click to enter: Smooth animated zoom into a node → it dissolves into its L2 children → breadcrumb updates
  • Continue drilling: Same pattern, L2 → L3 → L4
  • Navigate back: Click breadcrumb, press Escape, or click the dimmed parent border to zoom out

High L1/L2/L3 labels are insider jargon (mitigated by colour) UXBusiness

"L1", "L2", "L3" appear everywhere and are never explained. A project lead from Monday.com has no concept of "L3." However, the depth-colour gradient significantly mitigates this — users can visually distinguish levels through colour (blue = strategic, pink = atomic) even without understanding the labels. The jargon is still worth fixing, but it's less critical than initially assessed.

💡 Recommendation Allow users (or the AI) to name each level per project: "Phases → Workstreams → Tasks → Subtasks." Fall back to "Level 1", "Level 2" (not "L1"). The existing depth-colour gradient is excellent — keep it and ensure the named labels are displayed alongside the colour so users build the association: "blue = Phases, pink = Subtasks."

High No breadcrumbs for level navigation UXBusiness

When drilling from L1 → L2 → L3, the only wayfinding is a level badge and a back arrow. In a system where the entire value is hierarchical decomposition, users must always know where they are.

💡 Recommendation Persistent clickable breadcrumb below the header: Project > Strategy > Quick Wins > First Deployment. Each segment clickable. Show task count at current level. On narrow viewports, collapse to: ... > Quick Wins > L3.

High 3D multi-level overview needs interactivity UXUIFrontend

The 3D isometric layer diagram is visually impressive and conceptually bold — no competitor has this. But it's confined to a small overlay (~200×200px) with no visible click targets, labels, or interaction affordances. It reads as decoration, not navigation.

💡 Recommendation Promote it: Make it accessible via a persistent "Levels" button, not just a minimap corner. Make it interactive: Click a layer to navigate; hover to highlight and show tooltip ("Level 2: Workstreams — 8 tasks"). Add labels directly on each layer. Animate transitions when drilling — the selected level rises to the foreground. Assign each level a distinct colour (currently only L3 gets colour treatment).

Medium Swimlane view is overwhelming UXUIBusiness

Dense nodes with ~9px labels, duplicate names, no legend, and the lower two-thirds of the viewport empty. This is the most cognitively demanding screen in the product — even though the depth-colour gradient and circle/hexagon shapes help with orientation.

💡 Recommendation Use card-based nodes (consistent with kanban) instead of circles. Increase minimum label size to 12px. Suppress repeated metadata ("Unassigned" shown once at group level, not per node). Add a legend. Default new users to Canvas or List; surface Swimlane in "All Views."

Interaction Patterns for Level Navigation

🗺️ Google Maps Zoom

Scroll/pinch to zoom doesn't just scale — it transitions between levels. Zoom into an L1 node → it dissolves into L2 children. Makes level navigation a physical gesture rather than an abstract menu selection.

📁 Folder Mental Model

In list and kanban views, double-click an L1 item to expand its children. Clicking an L1 kanban card with children shows "View subtasks →". A persistent breadcrumb always shows the drill path.

✨ Animated Transitions

Level changes should be animated to maintain spatial context. Drilling down: current view zooms forward, new level fades in. Drilling up: reverse. The 3D panel should animate in sync.

👀 Hover Preview

Before committing to drill-down, hovering a node for 500ms shows a tooltip card: child count, status summary ("3 done, 2 blocked"), and "Click to explore." Gives confidence before navigating.

05 The Workspace — Human-AI Collaboration

The three-panel workspace (Output + Chat + Details) is where work gets done. The current implementation is more sophisticated than it first appears — the recommendations below aim to refine what exists, not propose basics that are already there.

✅ What Already Works — Workspace Capabilities

The workspace is considerably more capable than a basic three-panel layout. The Chat panel features expandable AI "Thought process" steps (showing multi-step reasoning chains), an "Execute this task for me" button, task execution history, and artifact creation status indicators. The Workspace panel has a rich text editor with formatting toolbar, a Preview toggle, tabbed outputs, and version navigation (V3/3) with arrow controls. The Details panel includes a History tab with a full timestamped audit trail ("Ali Panju assigned the task to Deepflow", "Ali Panju changed the task description"), a Resources tab with DeepFlow Inputs and Outputs sections, Sub-tasks, and an Advanced tab. This is genuine task execution infrastructure for running a business — not a toy.

High Cross-panel integration could be stronger UIUX

The individual panels are well-built, but the connections between them could be tighter. When the AI's thought process completes and creates an artifact, the workspace panel should visually acknowledge the new content. When a task fails, the error state should propagate across all three panels coherently — not just appear as a status badge in Details.

💡 Recommendation State coherence: When a task fails, the entire workspace should reflect it — subtle error tint, diagnostic view in the chat, and a "Retry" action accessible from any panel. Artifact bridging: When the chat shows "Creating artifact ✓", highlight or pulse the corresponding workspace tab. Thought process → output linking: Consider a subtle visual thread connecting the AI's reasoning steps to the generated content, reinforcing the "here's WHY this output looks this way" narrative.

High No graph context in task view Business

When drilling into a task, the graph disappears entirely. The user loses all sense of where this task sits in the workflow. The dependency structure — the product's core value — is invisible at the moment when it matters most. The depth-colour is present (the task header shows the level colour), but the dependency chain is not.

💡 Recommendation Add a mini dependency context bar at the top of the task view, colour-coded by depth level: ← Research (✓ Done) → [THIS TASK: Build Prototype (🔴 Failed)] → Testing (⏸ Blocked). Use circles/hexagons to indicate whether upstream/downstream tasks are human or AI — maintaining the shape language even in this compact view.

High Panel resize and collapse affordances are invisible UXFrontend

Three panels on a laptop screen (~1366px) will be tight. No visible resize handles, no collapse toggles.

💡 Recommendation Add visible drag handles between panels. Add collapse toggles (chevrons) on panel headers. Remember panel state per user. On screens < 1400px, default to two panels with Details as a slide-over. On mobile, single panel with tab bar.

Medium Version navigation could be more discoverable UX

The version navigation (V3/3 with arrows) is present but easy to overlook. Version history is powerful for business users who want to see how AI iterations evolved — but only if they notice it exists.

💡 Recommendation When multiple versions exist, add a subtle "3 versions — compare" label near the version arrows. Consider a diff view toggle that highlights what changed between versions. On first multi-version encounter, show a brief coach mark: "DeepFlow iterated on this — use arrows to see previous versions."

Medium Thought process disclosure could be richer UX

The expandable "Thought process" steps (Step 1, Step 2) are a great transparency feature. Business users running an entire operation on DeepFlow will want to understand why the AI made certain decisions — this is the mechanism for that trust.

💡 Recommendation Consider adding a summary line for collapsed thought processes: "2 steps · Prepared a French sonnet, then crafted it" — giving a quick overview without needing to expand. For complex multi-step reasoning, add a visual step indicator (dots or a mini progress bar) showing how far through the reasoning chain the AI progressed.

06 Making It Approachable

The non-technical user doesn't need a simpler product. They need a product that explains itself — with familiar shapes, human language, and a clear on-ramp to genuine power.

Onboarding Strategy

"A flowchart that runs itself." — the one-sentence mental model every business user can hold onto.

🎬 Step 1: The Build Animation

When a user types their goal, show the graph building itself — nodes appearing, connections forming, levels expanding. This 3-5 second animation teaches the core concept without a single word: you described a goal, and we structured it.

🏷️ Step 2: The Anatomy Overlay

On first graph load, a lightweight in-context annotation layer labels the key elements: "Each box is a task" • "Lines show dependencies" • "These run in parallel." Three annotations. Ten seconds. Dismissible.

🌱 Step 3: Start Tiny

The first project should be a 3-node graph: one starting task branching into two parallel tasks. Build confidence before complexity. The 90-task, 4-level project comes later.

🎯 Step 4: Learn by Doing

"Click a node → see the task → mark it done → watch the graph update." Contextual tooltips, celebration moments when dependencies resolve, "Why is this blocked?" right-click action. Teach through interaction.

Progressive Disclosure Strategy

  • Level 0 — The Prompt: User types a goal. No graph visible yet.
  • Level 1 — The Generated Plan: Clean, simplified graph with large nodes, clear labels, prominent status. L1 nodes only.
  • Level 2 — On-Demand Depth: Users who click a node see children. Users who never click stay at L1 — and that's fine.
  • Level 3 — Power Features: Dependency lines, critical path, swimlane view, edge labels — behind a "Show details" toggle.

Two User Postures

🤖 Auto-Pilot Mode

DeepFlow generates and executes autonomously. User monitors via the Level Hub dashboard — familiar KPI widgets, no graph required. Best for delegators and stakeholders who want oversight without direct manipulation.

✋ Manual Mode

User works directly with the graph — adding nodes, adjusting dependencies, reassigning tasks. Best for hands-on project leads who want control. Default new users to Auto-Pilot; surface editing tools progressively.

07 Visual Design System

The graph should be the most visually refined surface in the entire product. Every other view is a lens on graph data — the canvas should feel like the flagship experience.

Critical No unified component library UI

Button styles vary across every screen. Tab components appear in two incompatible patterns. Badge colours for level indicators differ between views. This signals "work in progress" rather than "production product."

💡 Recommendation Build a design system with exactly one variant of each component type. Audit every screen against it. A Storybook instance enforces consistency.

High Mixed icon families create visual noise UI

At least three icon styles coexist: outline/stroke (sidebar), filled (node badges), white-on-black (radial menu).

💡 Recommendation Migrate to a single icon family (Phosphor or Lucide). Define stroke weight, size grid (16/20/24px), and fill rules once.

High Hero banner wastes space for returning users UIUX

The rainbow gradient hero and display-size text pushes functional content below the fold. The gradient introduces off-palette colours not used elsewhere.

💡 Recommendation Collapse to a compact search/prompt bar for returning users (auto-collapse after 3+ projects). Align gradient to brand palette (cyan-to-deep-blue).

✅ The Depth-Colour System — A Foundation to Build On

DeepFlow's blue→red depth gradient is a genuinely sophisticated colour system. L1 strategic objectives render in blue/cyan, deepest atomic tasks in pink/red, with intermediate levels blending between. This gradient applies consistently across nodes, connection lines, badges, and the 3D navigator — creating a cohesive "you are here" signal at every level. Combined with the circle/hexagon shape distinction, DeepFlow already encodes two critical dimensions (depth and executor type) through visual properties alone. The recommendations below aim to preserve and extend this system, not replace it.

Medium Status needs a third visual dimension that doesn't conflict with depth-colour UI

Depth = colour (blue→red gradient). Executor = shape (circle vs hexagon). Status currently relies on small badge overlays. Adding status as a third independent visual dimension would make all three critical properties scannable simultaneously.

💡 Recommendation Use border style or glow for status: solid border for complete (with green glow), animated dashed border for in-progress, dotted/faded for pending, red pulse for failed. This keeps the depth-colour gradient as the fill/border colour, the shape as the executor signal, and the border treatment as the status signal — three independent channels, zero conflict. See the Style Guide for detailed specifications.

Design Principles for the Graph

🎨

Colour is Data (and Already Is)

The depth-colour gradient already makes colour meaningful. Extend this: every visual property should encode information. Shape = executor, colour = depth, border = status, size = importance. Eliminate decorative elements that don't carry meaning.

📖

Readability Over Aesthetics

A business user should understand workflow state in 3 seconds. Clear flow direction, depth-colour showing hierarchy at a glance, circle/hexagon showing the human/AI split, and status visible through border treatment — not just tiny badges.

🔎

Progressive Disclosure via Zoom

At far zoom: coloured shapes (circles and hexagons) tell depth and executor type. At medium zoom: add titles and status. At close zoom: full metadata. Beautiful and informative at every zoom level.

💫

Animation Communicates Life

Dot-flow along depth-coloured edges, gentle pulse on nodes needing attention, smooth level transitions with colour shifting. These separate "static diagram" from "living orchestration platform."

🌈

Respect the Gradient

The blue→red depth gradient is the primary colour system. UI chrome, status indicators, and decorative elements must not conflict with it. Use neutral greys for non-depth UI, and encode status through border style/glow rather than competing hues.

🔤

Typographic Discipline

Strict 5-level type scale (12px–28px). No 9px swimlane labels, no display-size hero text. Hierarchy through weight and colour, not dramatic size jumps.

10 Technical Feasibility

Implementation guidance for the recommended improvements, with library recommendations and performance strategy.

Recommended Architecture

🏗️ Graph: Single React Flow Instance with Virtual Layers

One canvas, one event system, one render loop. Nodes carry a level attribute. Drill-down applies a filter transition: fade out current level, animate viewport to children, fade in next level. The 3D overview panel (React Three Fiber) syncs as a read-only navigation minimap. This is simpler, more performant, and more maintainable than nested instances or 3D editing surfaces.

Recommended Stack

Graph Rendering

  • @xyflow/react v12 — Graph canvas with viewport culling
  • elkjs — Hierarchical layout in Web Worker
  • @react-three/fiber — 3D overview minimap only
  • d3-hierarchy — Edge bundling for dense graphs

Workspace

  • @tiptap/react — Rich text editor (ProseMirror-based)
  • react-resizable-panels — Panel layout with persistence
  • react-markdown — Streaming AI content rendering
  • shiki — Syntax highlighting (WASM)

State & Data

  • zustand + immer — Client state with undo/redo
  • @tanstack/react-query — Server state, caching, optimistic updates
  • EventSource (SSE) — Streaming AI responses

UI & Interaction

  • @radix-ui/* — Accessible primitives
  • @dnd-kit — Kanban drag & drop
  • framer-motion — Level transitions, animations
  • cmdk — Command palette for power users

Accessibility Requirements

  • Keyboard navigation: Tab into graph → Arrow keys between nodes (following edges) → Enter to drill down → Escape to drill up → / to search
  • Screen reader alternative: "View as accessible table" button switching to list view with dependency columns
  • Colour-blind safety: Shape encoding alongside colour — ✓ (done), ▶ (active), ○ (ready), ✕ (failed), ◻ (pending)
  • Reduced motion: Respect prefers-reduced-motion — disable animated transitions, dot-flow, and pulse effects

11 Implementation Roadmap

A phased approach over approximately 17 weeks, from quick wins through scale testing.

Phase 1

Make It Readable
