Core Thesis

Triggers and integrations are how DeepFlow connects to the real world. The architecture of how triggered execution runs are created, visualised, and managed determines whether DeepFlow feels like an automation tool or an orchestration platform.

DeepFlow's core engine — natural language to dependency graph, mixed human/AI execution, governance and audit — is already built and functional. But a powerful engine sitting in isolation is a tech demo. The moment DeepFlow responds to a Slack message, fires on a schedule, or reacts to a webhook from Stripe, it stops being a tool users visit and becomes infrastructure they rely on.

This document examines how that connection should work: what triggers an execution, where that execution lives in the product's information architecture, how users monitor and manage runs, and what the integration boundary looks like between DeepFlow and the services it connects to.

The stakes are high. Get this wrong and DeepFlow becomes another Zapier clone with a graph view bolted on. Get it right and the graph engine becomes the nervous system of an organisation's operations — reacting to signals, orchestrating responses, and maintaining a governed audit trail of everything that happened and why.

1. The Integration Landscape

What exists today

DeepFlow already has a working Slack integration. Users can interact with workflows from Slack, receive notifications, and trigger actions. Plans exist for email, Jira, and broader service connectivity. This isn't a blank page — it's a foundation to build on.

The Pipedream bet

Using Pipedream for authentication and custom tool orchestration is a genuinely smart architectural decision. OAuth is a graveyard of engineering time. Every integration needs token management, refresh flows, scope negotiation, and error handling. Pipedream abstracts all of this and provides a marketplace of pre-built connectors.

But the boundary matters. Where does Pipedream end and DeepFlow begin? If DeepFlow is just a fancy front-end for Pipedream workflows, users will eventually ask: "Why don't I just use Pipedream directly?" The answer needs to be clear: Pipedream handles auth and API plumbing. DeepFlow handles orchestration, governance, and the intelligence layer.

Think of it like a building: Pipedream is the wiring and plumbing. DeepFlow is the architecture, the floor plan, and the people working inside. Nobody confuses an electrician with an architect — the boundary should be equally obvious.

Categories of integration

CategoryDirectionExamplesDeepFlow Role
Input Triggers External → DeepFlow Slack message, inbound email, webhook, cron schedule, Stripe event Receives signal, creates execution run
Output Actions DeepFlow → External Send Slack message, update Jira ticket, write to database, send email, call API Task node executes external action as side-effect
Bidirectional Sync DeepFlow ↔ External Jira two-way sync, CRM data feed, real-time document collaboration Maintains state parity between systems

The first two categories are table stakes for v1. Bidirectional sync is complex (conflict resolution, staleness, eventual consistency) and should be deferred. Context enrichment is high-value but can be handled as a special case of output actions — the "output" is data fetched into the workflow context.

The integration abstraction layer

DeepFlow should present integrations to users as connections — not as API configurations. The mental model:

  • "I've connected my Slack workspace" — not "I've configured an OAuth2 client with scopes chat:write and channels:read"
  • "I've connected my Jira project" — not "I've set up a webhook endpoint at /api/v1/triggers/jira with basic auth"
  • "I've connected my email" — not "I've configured IMAP polling with a 60-second interval"

Pipedream handles the mechanical complexity. DeepFlow surfaces it as: Connected ✓ or Connect →. One button. One flow. Done.

┌─────────────────────────────────────────────────────────┐ │ DeepFlow Platform │ │ │ │ Orchestration Engine ←→ Trigger Router ←→ Run Manager │ │ ↕ ↕ │ │ Graph Executor Integration Layer │ │ ↕ │ ├─────────────────────────────────────────────────────────┤ │ Pipedream (Auth + API) │ │ │ │ Slack Gmail Jira Stripe GitHub HubSpot ... │ └─────────────────────────────────────────────────────────┘

The key principle: DeepFlow owns the intelligence. Pipedream owns the plumbing. Users never need to know Pipedream exists — they just see "DeepFlow connects to 500+ services."

2. Trigger Architecture

What is a trigger?

A trigger is an external event that initiates a workflow execution. It is the bridge between "something happened in the world" and "DeepFlow is now orchestrating a response." Every automated run in DeepFlow starts with a trigger. Manual runs — where a human clicks "Execute" — are themselves a trigger type.

Trigger taxonomy

TypeSignal SourceExampleContext Payload
Webhook HTTP POST from external service Stripe sends "payment_intent.succeeded" Full webhook body: customer ID, amount, currency, metadata
Schedule Cron expression / interval "Every weekday at 9:00 AM" generates daily report Timestamp, run sequence number, last run status
Message Chat platform event User types "/deepflow run weekly-standup" in Slack Channel, user, message text, thread ID, attachments
Email Inbound email matching filter Email to support@company.com triggers triage workflow Sender, subject, body, attachments, headers
Threshold Metric crosses defined boundary Error rate exceeds 5% → triggers incident response Metric name, current value, threshold, trend data
Manual Human clicks "Run" in DeepFlow UI Ravi clicks "Execute" on the ICAAP template User identity, any form inputs, selected parameters
Chained Another DeepFlow workflow completes Data pipeline finishes → triggers report generation Parent run ID, output data, completion status, duration

The trigger → execution lifecycle

Trigger fires │ ▼ Trigger Router receives event │ ├── Validate: Is this trigger active? Is the payload well-formed? ├── Authenticate: Does the source have permission? ├── Rate limit: Is this within acceptable frequency? │ ▼ Template Resolution │ ├── Look up the workflow template bound to this trigger ├── Resolve the current version of the template ├── Map trigger context → workflow input variables │ ▼ Execution Created │ ├── Instantiate a NEW run of the template graph ├── Inject trigger context as input variables ├── Assign execution identity (who does this run "as"?) ├── Record trigger metadata for audit trail │ ▼ Graph Executor begins processing │ ├── Root nodes with no dependencies → status: Ready ├── Agents/humans assigned → status: In Progress ├── Dependency propagation runs as normal │ ▼ Execution completes (or fails) │ ├── Notify trigger source (e.g., reply in Slack thread) ├── Fire chained triggers if configured └── Archive run with full audit trail

Trigger context: the seed data

This is a subtlety that matters enormously for UX. When a trigger fires, it carries context — and that context seeds the workflow execution. Without context mapping, every triggered run is identical. With it, each run is contextualised to the specific event that created it.

Example: A Slack trigger fires when someone posts in #support. The trigger context includes: user: "Sarah Chen", channel: "#support", message: "Client X can't access their dashboard". DeepFlow maps these to workflow inputs: reporter → Sarah Chen, issue_description → "Client X can't access their dashboard", priority → auto-detected from channel. The workflow template is the same every time. The execution is unique every time.

This mapping — trigger context → workflow inputs — is one of the most important configuration surfaces in the trigger system. It's also where the "automation tool vs orchestration platform" distinction crystallises. Automation tools map fields. Orchestration platforms understand context and let AI agents interpret it.

DeepFlow should do both: explicit field mapping for deterministic workflows, and AI context interpretation for workflows that benefit from intelligence at the input stage.

3. The Big Question: Where Do Execution Runs Live?

This is the most consequential architectural decision in the trigger system. Every triggered workflow creates a run — a living, breathing instance of a template graph. Where that run lives in DeepFlow's information architecture determines how users discover, monitor, and manage their automated operations.

There are three viable options. Each has genuine merit. Only one is right for DeepFlow.

Option A

Runs create new projects

Each trigger creates a new project in the gallery. The project gallery becomes the "runs" view.

  • Pro: Clean separation. Each run is fully self-contained. Full audit trail per run. Simple mental model — "every project is a run."
  • Pro: Works with the existing project-centric UI without major refactoring.
  • Con: The gallery becomes a junkyard. A daily report template running for 30 days creates 30 projects. Multiply by 10 templates and you have 300 projects in a month. The gallery ceases to be useful.
  • Con: No natural grouping. "Which of these 300 projects are daily reports?" requires naming conventions or tags — fragile human discipline.
  • Con: Ravi's personal command centre becomes a project dump. The exact opposite of "operating system."
Verdict: Works for low-frequency manual triggers. Catastrophic for automated/scheduled triggers at any scale.
Option B

Runs create tasks within a parent project

The template IS a project. Each trigger creates a new "instance" task (or sub-graph) within it. The project view shows all runs: completed, running, failed.

  • Pro: Natural grouping. "Daily Reports" is a project. Open it, see 30 runs. Intuitive.
  • Pro: Leverages existing project → task hierarchy. Less new UI surface.
  • Con: Conflates template definition with execution instances. The template's task structure (the graph definition) gets mixed in with hundreds of instantiated runs. Which nodes are "the template" and which are "run #47"?
  • Con: Status becomes ambiguous. Is the "Daily Reports" project 96% complete (because 29 of 30 runs succeeded) or is it ongoing (because it runs forever)?
  • Con: Depth hierarchy collision. DeepFlow uses depth (L1→L4) for task decomposition. Adding "runs" as another hierarchy level is semantically confusing — a run isn't a subtask of the template.
Verdict: Seductive in its simplicity but creates semantic confusion at scale. Template definition and execution history are fundamentally different concepts forced into one container.
★ Recommended — Option C

Runs are a separate concept — "Executions"

Templates live in the project gallery. Executions live in a separate "Runs" view. Like GitHub Actions: the workflow YAML is in the repo, runs are in the Actions tab.

  • Pro: Clean separation of definition and execution. Templates are clean. Runs are queryable, filterable, archivable.
  • Pro: Scales infinitely. 10,000 runs don't pollute the template gallery. The "Runs" view handles volume with filters, pagination, and search.
  • Pro: Natural for all three personas. Ravi sees "5 active runs" on his dashboard. Sarah filters by template and status to manage team operations. David filters by date range for audit.
  • Pro: Enables powerful run-level operations: compare runs, re-run with different parameters, bulk cancel, export audit for date range.
  • Con: Another navigation concept. Users need to learn: "Projects are templates. Runs are executions." Two places to look.
  • Con: Requires building a new "Runs" view that doesn't exist today.

Why Option C — for DeepFlow specifically

The GitHub Actions analogy is powerful because it's already understood. Developers (and increasingly non-developers) understand: "I have a workflow definition. It produces runs. Runs have statuses. I can re-run failed ones." This mental model maps perfectly to DeepFlow's template → triggered execution flow.

Consider the three personas:

  • Ravi (solo CEO) has 5-8 templates: daily standup, client onboarding, invoice processing, weekly report. He wants to see "3 runs active, 2 need attention" — not 300 projects. The Runs view gives him an operational dashboard. The project gallery stays clean — his 8 templates, curated and maintained.
  • Sarah (ops manager) manages a team running 20+ templates daily. She needs to answer: "What failed last night? Which runs are stuck? How many daily reports completed?" The Runs view with filters (template, status, date, assignee) gives her exactly this. Trying to answer these questions by digging through 600 projects would be unusable.
  • David (compliance) needs audit trails. "Show me all executions of the ICAAP workflow in Q4 2025, with their approval gate timestamps." A Runs view with structured metadata makes this a query. A project gallery makes this an archaeological expedition.

The navigation cost is real but manageable. Two top-level concepts: Templates (your workflow definitions) and Runs (your execution history). Both are linked — click "View Runs" from a template, or click "View Template" from a run. The relationship is bidirectional and always one click away.

4. Visualising Execution Runs

The "Live Runs" dashboard

When a user opens DeepFlow, they should immediately see: what's happening right now? Not what templates exist, not what was defined last week — what's running.

The Home dashboard should surface:

  • Active runs count — "7 workflows running" with a breakdown by template
  • Needs attention — runs that are stuck, failed, or waiting for human input. This is the priority queue. Red badge, prominent placement.
  • Recent completions — last 10 completed runs with status (success/failure), duration, and trigger source
  • Upcoming scheduled — next 5 scheduled triggers with countdown timers

Design principle: The dashboard should feel like an airport's flight departure board. At a glance: what's departed (completed), what's boarding (active), what's delayed (needs attention), what's next (scheduled). Not a spreadsheet — a status surface.

The Runs list view

The primary Runs view is a filterable, sortable list. Every run is a row.

ColumnContentWhy It Matters
StatusRunning / Completed / Failed / Waiting / CancelledThe first thing users scan for
TemplateName of the workflow templateGroups runs by purpose
TriggerWhat started it (Slack, schedule, webhook, manual)Context for "why did this run?"
StartedTimestamp with relative time ("2 min ago")Recency and ordering
DurationElapsed time or total time if completePerformance baseline
ProgressCompact bar: 12/18 nodes completeAt-a-glance progress without opening
Run AsIdentity/permissions of the executionAudit and access control

Filters: template, status, trigger type, date range, run-as identity. Saved filter presets for repeated queries ("Show me all failed ICAAP runs this quarter").

Run detail view

Clicking a run opens the instantiated graph — the actual execution, not the template. This is where DeepFlow's graph engine shines.

  • Live graph: Every node shows real-time status. Edges animate to show data flow. Completed nodes are green, active nodes pulse with cyan, failed nodes glow red. The user watches their workflow execute.
  • Node detail: Click any node to see: input data, output data, execution logs, duration, agent/human identity, and any errors.
  • Trigger context panel: A collapsible panel showing what triggered this run: the Slack message, the webhook payload, the schedule rule. Full provenance.
  • Timeline rail: A compact timeline at the bottom showing the run's execution over time. Hover to see which nodes were active at any moment. Useful for understanding parallelism and bottlenecks.

Failed run diagnostics

Failure handling is where orchestration platforms earn their keep. A failed task in a 40-node graph needs context, not just a red icon.

🔍 What the failure view should show
  • Which node failed — highlighted in the graph with a red glow and zoom-to-focus
  • Why it failed — error message, stack trace (for AI agents), or human's rejection reason
  • What was the input — the data that was fed to the failed node from its predecessors
  • Impact assessment — which downstream nodes are now blocked? How much of the graph is affected? Visualise the "blast radius" with a subtle red wash on all blocked dependents.
  • Recovery options — Retry this node. Skip and continue. Edit input and retry. Assign to different agent/human. Abort entire run.

The recovery options are critical. In a traditional automation tool, failure means "start over." In an orchestration platform, failure is localised. The 35 nodes that completed successfully don't need to re-run. The user fixes the one failure and the graph resumes. This is the DAG engine's superpower — and the run view must surface it clearly.

Comparing runs

Over time, users accumulate runs of the same template. Run comparison becomes valuable: "Why did Tuesday's report take 45 minutes when Monday's took 12?" A side-by-side view showing: node-level duration differences, different execution paths (if conditional), different input contexts. Not essential for v1, but the data model should support it from day one.

5. The Trigger Configuration UI

Setting up a trigger should feel like connecting a pipe — not like configuring a server. The goal is: 30 seconds from "I want this to run on a schedule" to "it's running on a schedule."

The configuration flow

Select a workflow template

Start from a template's settings page. The trigger is bound to a specific template — this is the "what" that will run.

Choose trigger type

Visual selector with icons: Webhook, Schedule, Slack Message, Email, Manual, Chained. Each shows a one-line description. "When a message is posted in a Slack channel." "On a recurring schedule." "When another workflow completes."

Configure trigger-specific settings

This panel changes based on trigger type:
• Schedule: Visual cron builder. "Every [weekday] at [9:00 AM] [London time]." No raw cron expressions.
• Slack: Select workspace (connected via Pipedream), choose channel, optional keyword filter.
• Webhook: Auto-generated URL. Copy it. Optionally set a secret for HMAC verification.
• Email: Auto-generated inbound address or filter rules on connected inbox.

Map trigger context to workflow inputs

The critical step. Left column: available trigger context fields ("Slack user name", "message text", "channel"). Right column: workflow input variables ("reporter", "task_description", "priority"). Drag to connect — or let AI suggest mappings. For simple cases, auto-map by name similarity.

Set execution permissions

Who does this run execute as? Options: the trigger creator's identity, the triggering user's identity (if applicable, e.g., the Slack user), or a service account. This matters for governance — the audit trail must record who authorised the execution.

Test the trigger

A "Send Test Event" button that simulates a trigger with sample data. The user sees the mapped inputs, confirms they look right, and watches a dry-run execute (or at least validates the first few nodes). This is the confidence moment — "yes, it works."

Where Pipedream auth fits

Steps 2 and 3 are where Pipedream's value materialises. When a user selects "Slack" as a trigger type, they need to authenticate their Slack workspace. Instead of DeepFlow implementing OAuth from scratch:

  1. User clicks "Connect Slack"
  2. Pipedream's auth popup appears (branded as DeepFlow via white-label or embedded)
  3. User authenticates once
  4. Pipedream stores the token, handles refresh
  5. DeepFlow receives a connection ID — that's all it needs

The user authenticated once. Every future Slack trigger reuses that connection. No re-auth. No token management visible to the user. This is the right UX.

Design consideration: The Pipedream auth popup must feel native. If it opens a Pipedream-branded page, users will be confused: "What is Pipedream? I thought I was using DeepFlow." Explore Pipedream's white-label options, or at minimum add a contextual message: "DeepFlow uses secure authentication via Pipedream to connect your accounts."

Multi-trigger templates

A single template can have multiple triggers. "Daily Report" might run on a schedule AND be manually triggerable AND respond to a Slack command. Each trigger is independently configured but creates runs of the same template. The trigger configuration UI should show all active triggers for a template in a list, with the ability to enable/disable, edit, or delete each one independently.

6. Agents as Triggers and Agents as Responders

This is where DeepFlow's architecture gets genuinely interesting — and where the "orchestration platform" identity becomes concrete.

Agents triggering workflows (chained execution)

When a task node in Workflow A completes, it can trigger Workflow B. This is the "Chained" trigger type from the taxonomy above. The implications are significant:

  • Pipeline orchestration: Data ingestion → Data validation → Report generation → Report distribution. Four separate templates, each triggered by the previous one's completion. Modular, reusable, independently debuggable.
  • Escalation patterns: Automated triage workflow identifies a critical issue → triggers the incident response workflow with the triage output as context.
  • Fan-out patterns: One workflow completes and triggers N workflows in parallel. "Quarterly review complete" → triggers "Update board deck", "File regulatory report", "Send client summaries" simultaneously.

The key design question: Is a chained execution a new, independent run — or a continuation of the parent run? The answer should be: a new, independent run with a linked parent. Each run has its own status, its own audit trail, its own failure handling. But the lineage is recorded: "This run was triggered by Run #4721 of 'Data Validation'." Lineage enables both debugging and compliance.

External events waking sleeping agents

Not every agent runs to completion in one burst. Some workflows have natural pause points: "Wait for client response", "Wait for data availability", "Wait for market close." When the external event occurs, the agent resumes.

This is architecturally different from a new triggered run. It's a resume signal, not a start signal. The distinction matters:

AspectTriggered Run (new)Resumed Agent (continuation)
Creates new graphYes — fresh instantiationNo — continues existing graph
StateClean slate + trigger contextFull prior state preserved
Audit trailNew run IDSame run ID, new entries
Trigger typeExternal eventExternal event matching a "wait" condition
UX metaphor"Starting a new job""Waking up and continuing work"

Both are valid. Both need to be supported. The UX must make the distinction clear. A "waiting" node in the graph should visually indicate what it's waiting for and show a clear "resume" action when the condition is met (or allow manual override).

Agent-to-agent communication within a run

Within a single execution run, one agent's output becomes another agent's input — this is just dependency resolution, which the DAG engine already handles. But there's a more interesting pattern: advisory signals. An agent working on Node 7 discovers something that's relevant to the agent on Node 12 (not a dependency, but useful context). Does DeepFlow support lateral communication between non-dependent nodes?

For v1: no. Lateral communication introduces complexity that isn't justified yet. Dependencies handle the critical path. Advisory signals can be modelled as shared context attached to the run (a "run context" that any node can read/write). But this is a v2+ consideration.

The meta-workflow: workflows about workflows

Once triggers and chaining exist, users will inevitably create "meta-workflows" — workflows that manage other workflows. Examples:

  • "Every Monday, check all scheduled workflows and report which ones failed last week"
  • "If any run exceeds its SLA duration, escalate to the team lead"
  • "When 3 runs of the same template fail consecutively, disable the trigger and alert the admin"

This is the orchestration platform reaching maturity: it orchestrates itself. DeepFlow doesn't need to build special-case features for these — if the trigger system is expressive enough and the run data is queryable, users will compose these patterns naturally.

Self-Critique

Is the trigger model too complex for v1?

Honest answer: yes, if you ship everything described above.

The full trigger system — seven trigger types, context mapping, execution permissions, chained workflows, agent resume, threshold monitoring — is a 6-month engineering effort at minimum. That's not a v1. That's a v3.

The MVP trigger system

Ship these three trigger types and nothing else:

  1. Manual — user clicks "Run" with optional parameter form. Zero new infrastructure. Validates the entire Runs view and execution lifecycle.
  2. Schedule — cron-based execution. Proves the trigger → template instantiation pipeline works. Most immediately useful for Ravi's "daily report" use case.
  3. Webhook — auto-generated URL, accepts POST. Enables Slack (via Slack's outgoing webhooks), Stripe, GitHub, and any service with webhook support. Maximum flexibility with minimum implementation.

These three cover 80%+ of trigger use cases. Slack-specific triggers, email triggers, and chained execution can follow once the foundation is proven. Context mapping can start as "pass the entire payload as JSON" and evolve to a visual mapper.

The Pipedream dependency: vendor lock-in risk

Pipedream is a venture-backed startup. Venture-backed startups can: raise prices, pivot, get acquired, or shut down. Each of these risks is real.

Mitigation strategy:

  • Abstraction layer. DeepFlow should never call Pipedream APIs directly from application code. An integration service with a clean interface (connect, disconnect, execute action, receive event) means Pipedream can be swapped for an alternative (Nango, Paragon, or a custom implementation) without touching the rest of the codebase.
  • Token portability. Understand where tokens live. If Pipedream stores OAuth tokens, can they be exported? If not, a migration means re-authenticating every user's connections. That's painful but survivable.
  • Webhook independence. Webhook triggers don't need Pipedream at all. If DeepFlow accepts HTTP POSTs directly, the simplest and most flexible trigger type is vendor-independent from day one.

The risk is manageable. The alternative — building OAuth for 500 services — is a years-long distraction from the core product. Pipedream is the right bet for now, with the abstraction layer as insurance.

Where does "workflow orchestration" end and "iPaaS" begin?

This is the existential question. iPaaS (Integration Platform as a Service) tools — Zapier, Make, Workato, Tray.io — focus on connecting systems: "when X happens in System A, do Y in System B." Workflow orchestration platforms focus on managing work: "here's a complex process involving multiple steps, people, and agents — execute it with governance."

DeepFlow should not become an iPaaS.

The temptation is real: once you have triggers and integrations, users will ask for data transformation, conditional routing, API mapping, error retry policies, response parsing — the full iPaaS feature set. Each request is reasonable in isolation. In aggregate, they transform DeepFlow from an orchestration platform into an integration tool.

The boundary rule: DeepFlow orchestrates work. It delegates integration to Pipedream (or equivalent). If a feature is about connecting System A to System B with data transformation, it belongs in the integration layer. If a feature is about coordinating a multi-step process with humans and AI agents, it belongs in DeepFlow.

Concretely:

  • DeepFlow's job: "When a support email arrives, triage it using AI, create a ticket, assign it to the right team member, track resolution, and notify the customer when done."
  • Pipedream's job: "Receive the email via IMAP, parse the headers, authenticate to Jira, create the ticket via API, format the Slack notification."

DeepFlow is the brain. Pipedream is the nervous system. Users interact with the brain. The nervous system operates invisibly.

Open questions for the team

Question 01

Should runs be immutable?

Once a run starts, can its graph be modified (add/remove nodes)? Or is the instantiated graph frozen — if you want a different structure, create a new run? Immutability simplifies audit trails but reduces flexibility.

Question 02

What happens to run data at scale?

1,000 runs per month × 40 nodes per run × logs and outputs per node = significant storage. Archival strategy? Retention policies? Can users export/delete old runs? This needs a data lifecycle plan.

Question 03

How do permissions work across triggers?

If Ravi sets up a webhook trigger, and an external service fires it, the run executes with Ravi's permissions. But what if Ravi leaves the company? Service account model? Trigger ownership transfer? This needs clear governance.

Question 04

Real-time or polling for the Runs dashboard?

WebSocket-based real-time updates for run progress, or polling? Real-time is a better UX (watching nodes complete live is compelling) but adds infrastructure cost. The graph engine may already support real-time — confirm and leverage.

The trigger and integration system is what transforms DeepFlow from a powerful graph engine into a platform that runs real operations. The MVP is deliberately narrow — manual, schedule, webhook — because the foundation (template → run separation, execution lifecycle, run visualisation) matters more than trigger variety. Build the Runs view first. Make it excellent. Then open the floodgates to every trigger type and integration the market demands.

The graph engine is built. The governance layer is built. Now connect it to the world.
